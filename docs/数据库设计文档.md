## 2. 表结构

### 2.1 `categories`

游戏类型表。当前 SQLite 文件中的字段顺序以 `PRAGMA table_info(categories)` 为准。

| 字段 | 类型 | 约束 | 中文描述 |
| --- | --- | --- | --- |
| `id` | INTEGER | PK | 类型主键 |
| `name` | TEXT | NOT NULL | 类型名称，例如 `动作`、`策略`、`仙侠` |
| `slug` | TEXT | NOT NULL, UNIQUE | 类型稳定标识，用于 API 类型筛选 |
| `description` | TEXT | NOT NULL | 类型说明，当前导入数据可为空字符串 |
| `is_visible` | INTEGER | NOT NULL, DEFAULT 1 | 是否在首页类型筛选区展示，`1` 表示展示，`0` 表示隐藏 |
| `game_count` | INTEGER | NOT NULL, DEFAULT 0 | 当前关联游戏数量，根据 `game_categories` 统计维护 |

显示规则：

- 根据 `game_categories` 统计每个类型关联的去重游戏数。
- 统计结果写入 `categories.game_count`。
- 关联游戏数小于 `10` 时，`is_visible = 0`。
- 关联游戏数大于等于 `10` 时，`is_visible = 1`。
- `/api/categories` 只返回 `is_visible = 1` 的类型。

### 2.2 `games`

游戏主表。当前 SQLite 文件中的字段顺序以 `PRAGMA table_info(games)` 为准。

| 字段 | 类型 | 约束 | 中文描述 |
| --- | --- | --- | --- |
| `id` | INTEGER | PK | 游戏主键 |
| `title` | TEXT | NOT NULL | 游戏名称 |
| `slug` | TEXT | NOT NULL, UNIQUE | 游戏稳定标识，用于详情页 URL |
| `category_id` | INTEGER | NOT NULL, FK -> `categories.id` | 主类型 ID，兼容旧单类型字段，当前取游戏类型列表中的第一个类型 |
| `studio` | TEXT | NOT NULL | 工作室或发行方，当前导入数据多为空字符串 |
| `release_year` | INTEGER | NOT NULL | 发行年份，无法解析时为 `0` |
| `rating` | REAL | NOT NULL | 评分，当前导入数据默认 `0.0` |
| `cover_url` | TEXT | NOT NULL | 封面地址，导入自 CSV 的 `游戏封面` 字段，缺省使用默认封面 |
| `summary` | TEXT | NOT NULL | 游戏简介，导入自 CSV 的 `游戏描述` 字段，缺省按基础字段生成 |
| `details` | TEXT | NOT NULL | 旧详情字段，当前保留用于兼容，通常为空字符串 |
| `platforms` | TEXT | NOT NULL | JSON 字符串，平台列表，当前默认包含 `PC` |
| `tags` | TEXT | NOT NULL | JSON 字符串，标签列表，当前包含游戏类型和资源目录名 |
| `created_at` | TEXT | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| `download_url` | TEXT | NOT NULL, DEFAULT '' | 游戏下载地址 |
| `size` | TEXT | NOT NULL, DEFAULT '' | 游戏大小，导入自 CSV 的 `大小` 字段，无法识别时为空字符串 |

字段说明：

- `category_id` 是兼容字段，不代表游戏只有一个类型。
- 游戏真实的全部类型以 `game_categories` 为准。
- 前端展示大小时读取 `games.size`。
- 前端展示下载地址时读取 `games.download_url`。
- 前端封面为空或加载失败时统一回退到 `frontend/public/assets/covers/default-game-cover.jpg`。

### 2.3 `game_categories`

游戏与类型的多对多关系表。

| 字段 | 类型 | 约束 | 中文描述 |
| --- | --- | --- | --- |
| `game_id` | INTEGER | PK, FK -> `games.id` | 游戏 ID |
| `category_id` | INTEGER | PK, FK -> `categories.id` | 类型 ID |
| `sort_order` | INTEGER | NOT NULL, DEFAULT 0 | 类型展示顺序，越小越靠前 |

关系说明：

- 一个游戏可以对应多个类型。
- 一个类型可以对应多个游戏。
- 删除游戏时，关联类型关系自动删除。
- 删除类型时，关联关系自动删除。
- 前端展示类型时按 `sort_order` 和类型名称排序后的结果拼接。

### 2.4 `ads`

广告位预留表。当前前端不展示广告布局，但后端保留查询能力。

| 字段 | 类型 | 约束 | 中文描述 |
| --- | --- | --- | --- |
| `id` | INTEGER | PK | 广告主键 |
| `placement` | TEXT | NOT NULL | 广告位标识，例如 `home_top`、`sidebar` |
| `title` | TEXT | NOT NULL | 广告标题 |
| `description` | TEXT | NOT NULL | 广告描述 |
| `image_url` | TEXT | NOT NULL | 广告图片地址 |
| `target_url` | TEXT | NOT NULL | 点击跳转地址 |
| `is_active` | INTEGER | NOT NULL, DEFAULT 1 | 是否启用，`1` 表示启用 |

当前保留广告位：

| placement | 说明 |
| --- | --- |
| `home_top` | 首页顶部广告位预留 |
| `sidebar` | 详情页侧边栏广告位预留 |

已移除广告位：

| placement | 原用途 | 当前状态 |
| --- | --- | --- |
| `forum_feed` | 论坛信息流广告位 | 已删除 |

## 3. 关系图

```text
categories 1 ── n game_categories n ── 1 games

ads 为独立预留表
```

说明：

- `games.category_id` 仍保留为主类型兼容字段。
- 当前业务查询和展示以 `game_categories` 多类型关系为准。

## 4. 类型筛选规则

后端 `/api/games` 类型筛选使用 `game_categories` 关系表。

- 支持传 `categories.slug`。
- 支持传 `categories.name`。
- 支持逗号分隔或重复参数传多个类型。
- 多个类型之间是 OR 关系。
- 不使用 `LIKE` 做类型匹配。
- `/api/categories` 仅返回 `is_visible = 1` 的类型，用于前端类型筛选展示。
- `/api/categories` 按 `game_count DESC, name ASC` 排序。

示例：

```http
GET /api/games?category=策略,仙侠&page=1&page_size=48
```

## 5. 数据导入规则

当前核心数据由 `tools/import_fufu_quark_games.py` 从 `data/fufu_quark_only.csv` 导入。

CSV 字段映射：

| CSV 字段 | 数据库字段 | 说明 |
| --- | --- | --- |
| `subfolder` | `games.title` | 游戏名称 |
| `url_or_filename` | `games.download_url` | 游戏下载地址 |
| `大小` | `games.size` | 游戏大小 |
| `年份` / `发布时间` | `games.release_year` | 发行年份，优先使用年份字段 |
| `游戏描述` | `games.summary` | 游戏简介 |
| `游戏封面` | `games.cover_url` | 游戏封面地址 |
| `游戏类型` | `categories`、`game_categories` | 按 `|` 拆分后生成多类型关系 |
| `root_dir_name` | `games.tags` | 作为资源目录标签写入 |

导入后会刷新：

- `categories.game_count`
- `categories.is_visible`
- `game_categories`

## 6. 迁移要求

`initialize_database()` 会执行轻量迁移：

- 创建当前核心表。
- 补充 `games.download_url` 字段。
- 补充 `games.size` 字段，并从旧 `details` / `summary` 文本中的 `资源大小` 回填。
- 补充 `game_categories` 关系表。
- 补充 `categories.game_count` 字段。
- 补充 `categories.is_visible` 字段。
- 根据类型关联游戏数刷新分类数量和展示状态。
- 清理旧初始硬编码游戏。
- 删除已移除的账户和论坛表。
- 删除 `forum_feed` 广告位。

## 7. 静态导出规则

GitHub Pages 不能运行 SQLite，因此需要通过 `tools/export_static_game_data.py` 导出静态数据。

导出目标：

```text
frontend/public/game-data.js
```

导出内容：

- 可见类型列表。
- 全部游戏列表。
- 游戏全部类型。
- 游戏详情。
- 下载地址。
- 游戏大小。
- 平台和标签。
- 封面地址。
