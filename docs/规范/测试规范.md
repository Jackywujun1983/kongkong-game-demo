# 测试规范

本文档作为当前项目及后续项目的默认测试规范，适用于 Python 后端、前端应用、接口服务、端到端流程和持续集成。测试代码必须同时遵守项目内的编码规范、前端开发规范和本文档要求。

## 1. 基本原则

- 测试应验证用户价值、业务规则和稳定接口，而不是复制实现细节。
- 每个重要功能都应覆盖正常路径、边界条件和关键异常路径。
- 测试应稳定、可重复、可自动化运行。
- 测试失败应能明确指出问题位置和失败原因。
- 测试数据应独立、可控、可清理。
- 不为了覆盖率编写无意义测试。
- 修复缺陷时应优先补充回归测试。

## 2. 测试分层

推荐采用分层测试策略：

| 层级 | 目标 | 常见工具 | 执行频率 |
| --- | --- | --- | --- |
| 单元测试 | 验证函数、类、组件内部逻辑 | `pytest`、`unittest`、`Vitest` | 每次提交 |
| 组件测试 | 验证 UI 组件在用户视角下的行为 | Testing Library、Vitest | 每次提交 |
| 接口测试 | 验证 API 请求、响应、权限和错误码 | pytest、Playwright API、HTTP 客户端 | 每次提交或每日 |
| 集成测试 | 验证模块之间协作 | pytest、Vitest、测试数据库 | 每次提交或每日 |
| 端到端测试 | 验证真实用户核心流程 | Playwright、Cypress | 主干合并、发布前 |
| 非功能测试 | 验证性能、安全、可访问性等 | Lighthouse、axe、压力测试工具 | 发布前或专项 |

测试数量建议：

- 单元测试和组件测试数量最多。
- 接口测试覆盖核心契约。
- 端到端测试只覆盖关键业务路径，避免用 E2E 替代所有测试。

## 3. 命名规范

### 3.1 Python 测试

- 测试文件使用 `test_*.py`。
- 测试函数使用 `test_` 前缀。
- 测试类使用 `*Tests` 或 `Test*`。
- 测试名称应描述行为和预期结果。

```python
def test_login_returns_token_when_credentials_are_valid() -> None:
    ...
```

### 3.2 前端测试

- 测试文件使用 `*.test.ts`、`*.test.tsx`、`*.spec.ts` 或 `*.spec.tsx`。
- 组件测试优先与组件同目录，或放在同模块 `__tests__/` 中。
- 测试名称使用用户行为描述。

```ts
it("shows error message when login fails", async () => {
  ...
});
```

### 3.3 端到端测试

- E2E 测试文件建议放在 `e2e/` 或 `tests/e2e/`。
- 文件名应对应业务流程。

```text
auth.spec.ts
game-search.spec.ts
forum-post.spec.ts
```

```text

```

## 5. Python 后端测试规范

后端测试推荐使用 `pytest`；如果项目暂无外部依赖，也可以使用标准库 `unittest`。当前项目已有标准库 `unittest` 测试，可在引入 `pytest` 后平滑迁移。

### 5.1 单元测试

适合覆盖：

- 纯函数。
- 数据转换。
- 参数校验。
- 权限判断。
- 密码哈希和 token 生成逻辑。

要求：

- 单元测试不依赖真实网络。
- 单元测试不依赖生产数据库。
- 外部依赖应使用 mock、fake 或测试替身。

### 5.2 仓储与数据库测试

数据库测试应使用临时数据库或独立测试数据库。

```python
def test_search_games_returns_matching_result(tmp_path: Path) -> None:
    database_path = tmp_path / "test.sqlite3"
    initialize_database(database_path)
    insert_test_game(database_path, title="测试星图边境")

    result = search_games(query="星图", database_path=database_path)

    assert result["total"] >= 1
```

要求：

- 不使用生产数据库。
- 每个测试应能独立运行。
- 测试结束后应清理数据或使用临时目录自动清理。
- SQLite 连接必须及时关闭，避免 Windows 文件句柄占用。

### 5.3 API 测试

接口测试应覆盖：

- HTTP 方法。
- 路径和查询参数。
- 请求体校验。
- 响应结构。
- 状态码。
- 鉴权和未授权场景。
- 错误消息。

示例场景：

- `GET /api/health` 返回服务状态。
- `GET /api/games` 支持关键词和分类。
- `POST /api/auth/login` 登录成功返回 token。
- 未登录访问发帖接口返回 `401`。

## 6. 前端测试规范

前端测试推荐使用 Vitest + Testing Library。

### 6.1 单元测试

适合覆盖：

- 工具函数。
- API 参数拼接。
- 状态派生。
- 数据格式化。

要求：

- 测试输入和预期输出应清晰。
- 不测试浏览器已保证的能力。
- 不测试样式细节，除非样式变化影响业务状态。

### 6.2 组件测试

组件测试应站在用户视角编写。

优先使用：

- 可访问名称。
- 文本。
- 表单 label。
- role。

避免优先使用：

- 深层 DOM 选择器。
- 组件内部状态。
- 纯 class 选择器。

示例：

```ts
it("opens game detail when user clicks detail button", async () => {
  ...
});
```

组件测试应覆盖：

- 加载状态。
- 空状态。
- 错误状态。
- 用户点击。
- 表单输入和提交。
- 权限相关 UI。

### 6.3 API Mock

前端测试不应依赖真实后端服务。

推荐：

- 使用 Mock Service Worker 或 Vitest mock。
- Mock 数据放在测试 fixture 中。
- 每个测试显式声明所需响应。
- 测试结束后重置 mock 状态。

## 7. 端到端测试规范

端到端测试推荐使用 Playwright。

### 7.1 覆盖范围

E2E 只覆盖核心用户流程：

- 进入首页并看到游戏列表。
- 搜索游戏并查看详情。
- 切换分类。
- 用户登录。
- 登录后发布论坛帖子。
- 未登录用户无法发布帖子。

### 7.2 定位器规范

优先使用稳定、贴近用户的定位方式：

- `getByRole`
- `getByLabel`
- `getByText`
- `getByPlaceholder`
- `getByTestId`

避免使用：

- 复杂 CSS 层级选择器。
- 依赖布局位置的选择器。
- 随样式变化容易失效的选择器。

### 7.3 断言规范

- 使用自动等待的断言。
- 不使用固定 `sleep` 等待。
- 对页面状态、文本、可见性和 URL 做明确断言。
- 每个测试只验证一个主要业务流程。

### 7.4 数据准备

推荐：

- 通过 API 或测试数据库准备数据。
- 测试账号与生产账号隔离。
- 每次测试使用唯一数据或执行清理。
- 不依赖测试执行顺序。

## 8. 接口契约规范

接口测试应保证前后端契约稳定。

必须关注：

- 字段名称。
- 字段类型。
- 必填字段。
- 错误响应格式。
- 分页结构。
- 鉴权 Header。

当前项目响应结构示例：

```json
{
  "items": [],
  "page": 1,
  "page_size": 12,
  "total": 0
}
```

错误响应结构：

```json
{
  "error": "Unauthorized"
}
```

接口字段变更时，应同步更新：

- 后端测试。
- 前端类型定义。
- 前端 API 封装。
- 技术文档。

## 9. 测试数据规范

- 测试数据应最小化，只保留验证场景所需字段。
- Fixture 名称应表达业务含义。
- 避免把大段业务无关数据复制到多个测试中。
- 不在测试数据中写入真实手机号、身份证、Token、密码等敏感信息。
- 密码类测试数据只用于测试环境。

推荐：

```text
tests/fixtures/
frontend/src/test/fixtures.ts
```

## 10. Mock 与测试替身

可以 mock：

- 网络请求。
- 时间。
- 随机数。
- 文件系统。
- 第三方服务。
- 邮件、短信、支付、广告等外部系统。

不建议 mock：

- 被测函数本身。
- 过多内部实现细节。
- 已经足够轻量且稳定的纯函数。

Mock 应在测试结束后恢复，避免污染其他测试。

## 11. 覆盖率规范

覆盖率是质量参考，不是唯一目标。

建议目标：

| 项目类型 | 行覆盖率 | 分支覆盖率 |
| --- | --- | --- |
| 核心业务模块 | 80% 以上 | 70% 以上 |
| 工具函数 | 85% 以上 | 75% 以上 |
| UI 展示组件 | 视风险决定 | 视风险决定 |
| 原型项目 | 可低于正式项目，但核心流程必须覆盖 | 可低于正式项目 |

覆盖率不足时，应优先补：

- 业务规则。
- 权限判断。
- 错误处理。
- 数据边界。

## 12. 缺陷回归规范

修复缺陷时必须记录：

- 缺陷触发条件。
- 期望行为。
- 实际行为。
- 修复点。
- 回归测试。

回归测试应能在缺陷修复前失败，在修复后通过。

## 13. CI 与提交前检查

提交前建议执行：

```bash
python -m compileall backend tools
cd backend
python -m unittest discover -s tests
```

当前项目若前端依赖已安装，额外执行：

```bash
cd frontend
npm run typecheck
npm run build
```

正式项目建议 CI 包含：

- Python 代码检查。
- Python 单元测试。
- 前端类型检查。
- 前端单元测试。
- E2E 核心流程。
- 构建检查。
- 覆盖率报告。

## 14. 稳定性要求

测试不得依赖：

- 测试执行顺序。
- 真实外部网络。
- 当前日期之外的隐式时间。
- 不固定的随机值。
- 生产环境账号。
- 上一次测试残留数据。

处理不稳定测试：

- 先定位原因，不直接提高重试次数掩盖问题。
- 对异步 UI 使用自动等待断言。
- 对时间和随机数进行控制。
- 对外部服务使用 mock 或测试环境。

## 15. 安全测试要求

至少覆盖：

- 未登录访问受保护接口。
- 普通用户访问管理员能力。
- 密码错误登录失败。
- 重复注册。
- 空输入和非法输入。
- 不泄露密码哈希、Token 等敏感字段。

后续扩展可增加：

- XSS 输入测试。
- CSRF 防护测试。
- SQL 注入边界测试。
- 文件上传安全测试。

## 16. 性能测试要求

性能测试应优先关注用户可感知路径：

- 首页加载。
- 游戏检索。
- 游戏详情打开。
- 登录。
- 发帖。

原型阶段可使用手动观察和轻量脚本。生产阶段建议增加：

- API 响应时间基线。
- 前端构建体积分析。
- Lighthouse 检查。
- 压力测试。

## 17. 可访问性测试要求

前端功能应关注：

- 按钮和链接语义正确。
- 表单具备 label。
- 图片具备 alt。
- 键盘可操作。
- 文本与背景对比度足够。
- 错误提示可被用户明确感知。

可使用 Testing Library、Playwright 和 axe 等工具辅助验证。

## 18. 当前项目测试清单

空空如也GameHub 当前版本至少应覆盖：

- 初始硬编码游戏不会被初始化。
- 分类列表初始化。
- 广告位初始化。
- 游戏关键词检索。
- 游戏分类检索。
- 多类型 OR 筛选。
- 游戏详情查询。
- 已移除账户、会话、论坛表不再初始化。
- 已移除账户和论坛接口返回不可用。
- 前端搜索交互。
- 前端分类切换。
- 前端分页交互。
- 前端详情页跳转。
