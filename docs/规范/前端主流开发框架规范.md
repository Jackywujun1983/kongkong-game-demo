# 前端主流开发框架规范

本文档作为项目前端开发的通用规范，适用于 React、Vue、Angular、Next.js、Nuxt、Svelte 等主流前端框架。具体项目应优先遵循当前技术栈的官方推荐写法，并在团队内保持一致。

## 1. 基本原则

- 优先保证可读性、可维护性和一致性。
- 组件职责清晰，避免一个组件承担过多逻辑。
- 状态流向明确，避免隐式共享和难以追踪的副作用。
- 页面、组件、样式、接口、测试应有稳定的组织方式。
- 优先使用框架生态中的成熟方案，避免重复造轮子。
- 对用户可见的功能应关注加载、错误、空状态和交互反馈。

## 2. 推荐技术组合

| 方向 | 常见选择 |
| --- | --- |
| 构建工具 | Vite、Webpack、Rspack、Turbopack |
| 语言 | TypeScript |
| UI 框架 | React、Vue、Angular、Svelte |
| 应用框架 | Next.js、Nuxt、Angular、SvelteKit |
| 状态管理 | Redux Toolkit、Zustand、Pinia、Vuex、NgRx |
| 请求管理 | TanStack Query、SWR、Axios、Fetch API |
| 样式方案 | CSS Modules、Tailwind CSS、Sass、CSS-in-JS |
| 代码检查 | ESLint |
| 格式化 | Prettier |
| 测试 | Vitest、Jest、Testing Library、Playwright、Cypress |

## 3. 项目结构

通用推荐结构：

```text
src/
├── assets/
├── components/
├── config/
├── hooks/
├── layouts/
├── pages/
├── routes/
├── services/
├── stores/
├── styles/
├── types/
├── utils/
└── main.ts
```

说明：

- `components/` 放通用组件。
- `pages/` 或 `views/` 放页面级组件。
- `services/` 放接口请求和外部服务封装。
- `stores/` 放全局状态管理。
- `hooks/` 或 `composables/` 放可复用逻辑。
- `types/` 放公共类型定义。
- `utils/` 放纯工具函数，避免堆积业务逻辑。

大型项目可按业务模块组织：

```text
src/
├── features/
│   ├── user/
│   │   ├── components/
│   │   ├── services/
│   │   ├── stores/
│   │   └── types.ts
│   └── order/
├── shared/
└── app/
```

## 4. 命名规范

| 类型 | 命名方式 | 示例 |
| --- | --- | --- |
| 组件 | 大驼峰 | `UserCard.tsx`、`UserCard.vue` |
| 页面组件 | 大驼峰或框架路由约定 | `UserListPage.tsx` |
| 普通变量 | 小驼峰 | `userName` |
| 函数 | 小驼峰 | `getUserList()` |
| 常量 | 全大写 + 下划线 | `MAX_RETRY_COUNT` |
| 类型 / 接口 | 大驼峰 | `UserInfo`、`UserListProps` |
| CSS class | kebab-case | `user-card` |
| Hook | `use` 开头 | `useUserList()` |
| Vue composable | `use` 开头 | `useUserStore()` |

避免使用模糊命名：

```ts
// 不推荐
const data = await getData();

// 推荐
const userList = await getUserList();
```

## 5. TypeScript 规范

- 新项目默认使用 TypeScript。
- 公共函数、组件 Props、接口响应应有明确类型。
- 避免滥用 `any`，确实无法确定类型时优先使用 `unknown`。
- 类型定义尽量靠近业务模块，公共类型放入 `types/`。
- 不将后端响应类型与前端展示模型强行混用。

```ts
interface UserCardProps {
  user: UserInfo;
  onSelect?: (userId: string) => void;
}

function UserCard({ user, onSelect }: UserCardProps) {
  return <button onClick={() => onSelect?.(user.id)}>{user.name}</button>;
}
```

## 6. 组件规范

组件应保持职责单一：

- 展示组件负责 UI 展示。
- 容器组件负责数据获取、状态编排和业务逻辑。
- 通用组件不直接耦合具体业务接口。
- 页面组件负责组合布局和路由级逻辑。

组件 Props 应清晰、稳定：

```ts
interface SubmitButtonProps {
  loading?: boolean;
  disabled?: boolean;
  onClick: () => void;
}
```

避免组件过大。若一个组件同时包含复杂状态、复杂布局和多个业务分支，应拆分为更小的组件或提取逻辑。

## 7. React 规范

- 函数组件优先。
- 自定义 Hook 使用 `use` 前缀。
- 避免在渲染过程中执行副作用。
- `useEffect` 只处理副作用，不用于无必要的数据派生。
- 列表渲染必须提供稳定的 `key`。
- 复杂状态优先考虑 `useReducer` 或状态管理库。

```tsx
function UserList() {
  const { data, isLoading } = useUserList();

  if (isLoading) {
    return <Loading />;
  }

  return (
    <ul>
      {data.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

## 8. Vue 规范

- Vue 3 项目优先使用 Composition API。
- 组件文件建议使用 `<script setup lang="ts">`。
- Composable 使用 `use` 前缀。
- Props 和 Emits 应声明类型。
- 避免直接修改 Props。
- 复杂状态可使用 Pinia。

```vue
<script setup lang="ts">
interface Props {
  title: string;
}

defineProps<Props>();

const emit = defineEmits<{
  confirm: [];
}>();
</script>
```

## 9. Angular 规范

- 使用 Angular CLI 生成项目结构。
- 组件、服务、模块职责分明。
- 业务逻辑优先放入 Service。
- RxJS 流应合理命名，Observable 变量可使用 `$` 后缀。
- 订阅应注意释放，避免内存泄漏。
- 表单优先使用 Reactive Forms。

```ts
users$ = this.userService.getUsers();
```

## 10. 路由规范

- 路由路径使用小写和中划线。
- 页面级组件与路由保持清晰对应。
- 路由守卫只处理访问控制，不堆积业务逻辑。
- 动态路由参数应有校验。
- 需要 SEO 的页面应正确设置标题和元信息。

```text
/users
/users/:id
/order-detail/:orderId
```

## 11. 状态管理规范

优先级建议：

1. 组件内部状态。
2. 父子组件 Props / Events。
3. Context / Provide Inject。
4. 状态管理库。
5. 服务端缓存管理工具。

不要把所有状态都放入全局 Store。以下状态适合全局管理：

- 登录用户信息。
- 权限信息。
- 全局配置。
- 跨页面共享的业务状态。

服务端数据建议使用 TanStack Query、SWR 等工具管理缓存、重试、失效和刷新。

## 12. 接口请求规范

- 统一封装请求客户端。
- 统一处理基础 URL、超时、鉴权、错误码。
- 接口层不要直接耦合 UI 组件。
- 对接口响应做类型定义。
- 对用户可感知错误提供清晰反馈。

```ts
export async function getUserList(): Promise<UserInfo[]> {
  const response = await http.get<ApiResponse<UserInfo[]>>("/users");
  return response.data.data;
}
```

## 13. 样式规范

- 样式命名保持一致。
- 避免大范围全局样式污染。
- 组件样式应尽量局部化。
- 颜色、字号、间距、圆角等应使用设计变量。
- 响应式布局优先使用弹性布局、网格布局和断点变量。

推荐 CSS 变量：

```css
:root {
  --color-primary: #2563eb;
  --color-danger: #dc2626;
  --spacing-sm: 8px;
  --spacing-md: 16px;
  --radius-md: 8px;
}
```

## 14. UI 与交互规范

- 所有可点击元素应有明确交互状态。
- 表单提交应有 loading 状态。
- 请求失败应有错误提示。
- 数据为空应有空状态。
- 危险操作应有确认。
- 长列表应考虑分页、虚拟列表或懒加载。
- 移动端应保证触控区域足够大。

## 15. 可访问性规范

- 图片应提供有意义的 `alt`。
- 表单控件应有关联的 label。
- 按钮使用 `<button>`，链接使用 `<a>`。
- 不只依赖颜色表达状态。
- 弹窗、菜单、抽屉应考虑键盘操作和焦点管理。
- 文本与背景应保持足够对比度。

## 16. 性能规范

- 路由级页面可按需加载。
- 大型组件、图表、编辑器等可懒加载。
- 图片应压缩，并使用合适格式。
- 列表数据较大时使用分页或虚拟滚动。
- 避免无意义重复渲染。
- 对频繁触发的输入、滚动、窗口变化使用防抖或节流。
- 打包产物应定期分析体积。

## 17. 安全规范

- 不在前端代码中硬编码密钥。
- 不信任来自用户或后端的 HTML 字符串。
- 谨慎使用 `dangerouslySetInnerHTML`、`v-html`。
- 登录态、权限、敏感操作必须由后端最终校验。
- 避免在 URL、localStorage 中存放敏感信息。
- 第三方依赖应定期升级和检查漏洞。

## 18. 测试规范

推荐测试分层：

| 类型 | 工具 | 关注点 |
| --- | --- | --- |
| 单元测试 | Vitest、Jest | 工具函数、状态逻辑、组件逻辑 |
| 组件测试 | Testing Library | 用户视角下的组件行为 |
| 端到端测试 | Playwright、Cypress | 核心业务流程 |

测试命名应描述行为：

```ts
it("shows error message when submit fails", async () => {
  ...
});
```

## 19. 代码检查与格式化

推荐使用 ESLint + Prettier。

```json
{
  "scripts": {
    "lint": "eslint .",
    "format": "prettier --write .",
    "test": "vitest",
    "typecheck": "tsc --noEmit"
  }
}
```

提交前建议执行：

```bash
npm run lint
npm run typecheck
npm run test
```

## 20. Git 提交规范

推荐使用 Conventional Commits：

```text
feat: add user profile page
fix: handle login timeout
docs: update frontend guideline
refactor: simplify order store
test: add checkout flow tests
chore: update dependencies
```

常见类型：

- `feat`: 新功能
- `fix`: 修复问题
- `docs`: 文档变更
- `style`: 格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建、依赖、脚手架等杂项

## 21. 框架选择建议

- 偏组件生态和大型社区：React。
- 偏渐进式开发和上手成本：Vue。
- 偏企业级强约束和完整工程体系：Angular。
- 偏 SSR、SEO、全栈路由：Next.js 或 Nuxt。
- 偏轻量和编译优化：Svelte 或 SvelteKit。

框架选择应结合团队经验、业务复杂度、生态成熟度、长期维护成本和现有系统情况决定。

## 22. 总结

前端代码应保持：

- 结构清晰
- 组件可复用
- 状态可追踪
- 类型明确
- 样式可维护
- 交互完整
- 性能可控
- 测试覆盖核心流程
